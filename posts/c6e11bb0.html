Total Blocking Time(总阻塞时间)——译

发表于2022-10-06|更新于2022-10-06|性能优化TBT翻译
|阅读量:

原文地址

概述

总阻塞时间 (TBT) 是 2020 年推出的 Lighthouse 性能指标，用于量化页面对用户输入的加载响应。

用最简单的术语来说，TBT 衡量您的网页被阻止 的总时间，从而阻止用户与您的页面进行交互。

它是Web Vitals之一，是PageSpeed Insights 中 仅字段首次输入延迟 (FID) 指标的替代品。

总阻塞时间衡量什么？

据谷歌称，

“TBT 测量第一次内容绘制 (FCP) 和交互时间之间的总时间，其中主线程被阻塞足够长的时间以防止输入响应。”

基本上，浏览器使用所谓的主线程来解析 HTML、构造 DOM、执行 CSS 和 JavaScript、处理用户事件以及执行其他重要任务。

当这些任务中的任何一个运行时间超过 50 毫秒（也称为长任务）时，主线程被视为“阻塞”，因为浏览器无法中断正在进行的任务。

如果主线程被阻塞，您的页面将无法响应用户输入，例如屏幕点击、键盘按下或鼠标点击。

超过 50 毫秒的额外时间被视为该请求的单独阻塞时间。

所有这些阻塞时间的总和就是您页面的 Total Blocking Time。
任何超过 50 毫秒的任务都被视为长任务。上面的示例有 3 个长任务（红色）。

例如，上图中，主线程上有 5 个任务，其中 3 个是长任务，因为它们各自的持续时间超过 50 毫秒。每个 Long Task 的阻塞时间如下：

  • 任务 A - 170 毫秒
  • 任务 B - 20 毫秒
  • 任务 E - 95 毫秒

在这种情况下，TBT 是285ms。但是，如果主线程只有一个需要 500 毫秒的任务，那么 TBT 将是450 毫秒。

总阻塞时间与交互时间

交互时间 (TTI)是与您的页面交互性相关的另一个指标。TBT 和 TTI 相辅相成，但为您的页面体验提供了完全不同的视角。

当您的页面完全交互时，TTI 会发出信号；TBT 专门告诉您哪些 JavaScript 任务执行时间最长。

如果主线程至少 5 秒没有长任务，则 TTI 认为页面是完全交互的。

考虑以下场景：

  • A）三个 60 毫秒的任务分布在 5 秒的时间段内。
  • B) 一个 5 秒长的任务。

上述两种情况都会将 TTI 推后相同的数量。

场景 A 在整个页面加载过程中主要是交互式的，因为没有长任务占用浏览器太多时间。

然而，这两种情况对用户来说会感觉非常不同，因为第一种情况的 TBT 仅为30 毫秒，而第二种情况的 TBT 为4950 毫秒。

场景 A 在整个页面加载过程中主要是交互的，因为长任务不会占用浏览器太多时间，而场景 B 在浏览器忙于处理长任务时根本不可交互。

这使得 TBT 既是一个有见地的独立指标，也是一个有用的 TTI 配套指标。

总阻塞时间与首次输入延迟

如前所述，TBT 是 Web Vitals 之一的 First Input Delay (FID) 的替代品。FID 是一个仅限字段的指标，需要真实的用户数据来衡量。

这些真实的用户数据以Chrome 用户体验报告 (CrUX)的形式出现——谷歌从真实的 Chrome 用户那里收集的 Chrome 浏览器行为数据库， GTmetrix并未在其测试中使用这些数据。

GTmetrix 报告总阻塞时间代替首次输入延迟。

因此，GTmetrix 测试报告 TBT 代替 FID，因为它充当合适的代理并推荐相同的优化。

总阻塞时间对您的绩效得分的影响

作为 Web Vital 指标，TBT 占 Performance Score 的 30%，是最需要优化的指标。

这突显了我们更加关注用户体验以补充页面速度分析。

这对您来说意味着，优化您的 TBT 通常可以对您的网站的响应能力做出最有影响力的改进之一。

总阻塞时间的阈值

总阻塞时间测量当页面被阻止响应用户输入并以毫秒为单位显示结果时 FCP和TTI之间的总时间量。

TBT 阈值如下：

  • 好 - 这里无事可做= 150 毫秒或更短的 TBT。
  • 好的，但考虑改进= 150 到 224 毫秒之间的 TBT。
  • 比推荐的长= TBT 在 224 到 350 毫秒之间。
  • 比推荐的长得多= TBT 高于 350 毫秒。

如何提高总阻塞时间

想要提高总阻塞时间首先要找到是哪些因素导致总阻塞时间变长，我们可以打开GTmetrix工具对要进行测试的页面进行测试。

当完成测试后会生成报告，点击报告的结构（或者是Structure），然后点击该页面的TBT按钮会显示所有造成阻塞的原因，以及如何进行优化和不适用的优化。
